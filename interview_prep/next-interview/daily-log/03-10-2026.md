### Question 1 — First Transaction Per Policy

#### ❌ My Attempt

```sql
WITH ranked_transactions AS (
    SELECT policy_id, transaction_date, amount,
        ROW_NUMBER() OVER (PARTITION BY policy_id ORDER BY transaction_date DESC) AS rn
    FROM transactions
)
SELECT policy_id, transaction_date, amount
FROM ranked_transactions
WHERE rn = 1;
```

#### ⚠️ Issue Identified

Used `ORDER BY transaction_date DESC` — this returns the **latest** transaction, not the **first**.

**Example — Effect of DESC ordering:**

| transaction_date | rn (DESC)                             |
| ---------------- | ------------------------------------- |
| Jan 10           | 1 ← returned (WRONG — this is latest) |
| Jan 05           | 2                                     |
| Jan 01           | 3 ← should be returned (first)        |

#### ✅ Correct Solution

```sql
WITH ranked_transactions AS (
    SELECT policy_id, transaction_date, amount,
        ROW_NUMBER() OVER (
            PARTITION BY policy_id
            ORDER BY transaction_date ASC   -- ✅ ASC for first record
        ) AS rn
    FROM transactions
)
SELECT policy_id, transaction_date, amount
FROM ranked_transactions
WHERE rn = 1;
```

#### Problems Identified

| #   | Problem                                                    | Fix             |
| --- | ---------------------------------------------------------- | --------------- |
| 1   | `ORDER BY transaction_date DESC` returns latest, not first | Change to `ASC` |

#### 💡 Key Learning

> Whenever you see **time-based logic**, always ask yourself:
> **"Should the timeline move forward or backward?"**
> Usually the answer is `ORDER BY date ASC`

**What was correct:**

- ✔ ROW_NUMBER pattern — correct
- ✔ CTE usage — correct
- ✔ Filtering with `WHERE rn = 1` — correct
- ⚠ Only the ordering direction was wrong

**Score: 8/10**

### Question 2 — Gap Between Transactions

#### ❌ My Attempt

```sql
SELECT policy_id,
    transaction_date,
    DATEDIFF(transaction_date,
        LAG(transaction_date) OVER (PARTITION BY policy_id ORDER BY transaction_date DESC)
    AS days_since_last_transaction
FROM transactions;
```

#### ✅ Correct Solution

```sql
SELECT
    policy_id,
    transaction_date,
    DATEDIFF(
        transaction_date,
        LAG(transaction_date) OVER (
            PARTITION BY policy_id
            ORDER BY transaction_date    -- ✅ ASC (default) for correct timeline
        )
    ) AS days_since_last_transaction
FROM transactions;
```

**Expected Output:**

| policy_id | transaction_date | days_since_last_transaction |
| --------- | ---------------- | --------------------------- |
| P1        | Jan 01           | NULL                        |
| P1        | Jan 10           | 9                           |
| P1        | Jan 15           | 5                           |

#### Problems Identified

| #   | Problem                                            | Fix                                 |
| --- | -------------------------------------------------- | ----------------------------------- |
| 1   | Missing closing parenthesis on `DATEDIFF`          | Add `)` after `LAG(...)`            |
| 2   | `ORDER BY transaction_date DESC` reverses timeline | Remove `DESC` — use `ASC` (default) |

#### 💡 Key Learning

`LAG()` looks at the **previous row** — so the partition must be ordered chronologically (ASC) for the difference to make sense forward in time.

```sql
LAG(column) OVER (PARTITION BY group ORDER BY time_column)
-- Returns the value from the previous row in the ordered partition
```

> ✅ **Good sign noted:** Understood the concept of `LAG` even without prior exposure — shows strong analytical SQL thinking.

**Score: 7/10**

### Question 3 — Monthly Premium Per Policy

#### ✅ My Attempt (Perfect)

```sql
SELECT
    policy_id,
    DATE_TRUNC('month', transaction_date) AS month,
    SUM(premium) AS total_premium
FROM transactions

```

#### Why This Is Production-Ready

- ✔ Correct aggregation with `SUM()`
- ✔ Correct date truncation with `DATE_TRUNC()`
- ✔ Correctly grouped by both `policy_id` AND `DATE_TRUNC()` — many candidates forget this

> Many candidates forget to include `DATE_TRUNC()` in the `GROUP BY` clause — this would cause an error or incorrect results.

## 🏗️ Case Study 1 — Spark Data Skew

### Problem Statement

A single policy has 200M+ records in a transactions table. Query performance is very slow.

### My Answer

> _"If one policy has 200M records, partitioning and shuffling will be problematic as one partition will be huge. Spark will struggle performing operations because there will be one skewed policy. Solution: include other attributes like policy_id + timestamp for partitioning to create smaller partitions."_

---

### ✅ What I Got Right

**Problem identification — Score: 8/10**

Correctly identified: one partition becomes disproportionately large, causing one executor to run much longer than others (the **straggler problem**).

| Partition  | Rows |
| ---------- | ---- |
| policy_123 | 200M |
| policy_456 | 10K  |
| policy_789 | 12K  |

Spark waits for the slowest partition — the job is only as fast as the slowest executor.

**Spark reasoning — Score: 7.5/10**

Correctly identified that Spark distributes data based on partition keys via shuffle:

```
GROUP BY policy_id
→ Spark: Hash(policy_id) → routes all rows to one partition
→ policy_123 = 200M rows on one executor
→ Other executors sit idle
```

---

### ⚠️ Where I Was Incomplete — Solution (Score: 6/10)

**My solution (composite partitioning)** is partially correct but not the strongest answer. Adding `timestamp` still risks clustering if timestamps are also skewed.

### ✅ Standard Interview-Level Solutions

#### 1. Key Salting _(Most important — must know)_

Add a random salt to distribute the skewed key across multiple partitions:

```python
from pyspark.sql.functions import concat, lit, floor, rand

# Salt the large key into N sub-keys
n_salt = 10
df_large = df_large.withColumn(
    "salted_key",
    concat(col("policy_id"), lit("_"), (floor(rand() * n_salt)).cast("string"))
)

# Explode the small table to match salted keys
df_small = df_small.withColumn(
    "salt", explode(array([lit(i) for i in range(n_salt)]))
).withColumn(
    "salted_key",
    concat(col("policy_id"), lit("_"), col("salt").cast("string"))
)

result = df_large.join(df_small, "salted_key").drop("salt", "salted_key")
```

#### 2. Broadcast Join _(When one side is small)_

```python
from pyspark.sql.functions import broadcast

result = large_df.join(broadcast(small_df), "policy_id")
# Avoids shuffle entirely — small table sent to every executor
```

#### 3. Adaptive Query Execution — AQE _(Spark 3+)_

```python
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
# Spark automatically detects and splits skewed partitions
```

#### 4. Repartition

```python
df.repartition(500, "policy_id", "transaction_date")
# More partitions = smaller chunks per executor
```

---

### 💡 Key Learning — Spark Skew Toolkit

| Solution           | When to Use                                           |
| ------------------ | ----------------------------------------------------- |
| **Key Salting**    | Skewed aggregation or join on high-cardinality key    |
| **Broadcast Join** | One table is small enough to fit in memory            |
| **AQE (Spark 3)**  | General skew — let Spark handle it automatically      |
| **Repartition**    | Redistribute data more evenly before heavy operations |

**Overall Case Study Score: 7/10**

## 🏗️ Case Study 2 — Late Arriving Data

### Problem Statement

Transactions sometimes arrive in the pipeline after their actual business date. This causes incorrect analysis and reports.

### My Answer

> _"If a transaction arrives late it can result in incorrect analysis. We can use Kafka or messaging queue to keep the pipeline stable as it sends records in fixed periods of time. This exists between Source System to Clean Zone layer in Data Vault."_

---

### ✅ What I Got Right

**Problem identification — Score: 8/10**

Correctly identified the core impact:

| Impact                                | Example                                                                            |
| ------------------------------------- | ---------------------------------------------------------------------------------- |
| Incorrect aggregates                  | Jan 10 report runs on Jan 11 — Jan 10 transaction arrives Jan 12 → report is wrong |
| Missing data in incremental pipelines | Pipelines that only process "today's" data miss late records                       |
| Incorrect KPIs                        | Loss ratio, premium totals all affected                                            |

---

### ⚠️ Where I Was Incomplete

**Kafka answer — Score: 6/10**

Kafka helps with **streaming ingestion** but does NOT solve the late arriving data problem itself. Kafka ensures records are delivered reliably — it does not fix the case where a transaction occurred 3 days ago and was only sent today.

**Data Vault layer — Score: 6.5/10**

Late data is handled in the **Raw Vault layer**, not between source and Clean Zone.

Raw Vault captures both timestamps:

| Column                          | Description                                  |
| ------------------------------- | -------------------------------------------- |
| `EFFECTIVE_DATE` / `EVENT_DATE` | Actual business date when the event occurred |
| `LOAD_DATE`                     | When the record was loaded into the vault    |

This means a late-arriving record is still captured correctly — it just has a `LOAD_DATE` later than its `EFFECTIVE_DATE`.

---

### ✅ Standard Interview-Level Solutions

#### 1. Reprocessing Window _(Most practical)_

Instead of processing only today's data, always process a rolling window:

```sql
WHERE transaction_date >= CURRENT_DATE - INTERVAL '3 days'
```

#### 2. MERGE / Upsert Pattern

Late data updates or inserts into existing records:

```sql
MERGE INTO warehouse_table AS target
USING staging_table AS source
ON target.transaction_id = source.transaction_id
WHEN MATCHED THEN UPDATE SET ...
WHEN NOT MATCHED THEN INSERT ...;
```

#### 3. Watermarking _(Structured Streaming)_

Define how late data is allowed to be before being dropped:

```python
df.withWatermark("event_time", "2 days") \
  .groupBy(window("event_time", "1 day")) \
  .sum("amount")
```

#### 4. Idempotent Pipelines

Design pipelines so re-running them produces the same result — safe to reprocess:

```python
df.write.mode("overwrite").partitionBy("transaction_date").parquet("output/")
```

#### 5. Data Vault — Dual Timestamp

```sql
-- Raw Vault Satellite correctly handles late data
INSERT INTO SAT_TRANSACTIONS (
    HUB_POLICY_HK,
    LOAD_DATE,          -- When it arrived in the vault (today)
    EFFECTIVE_DATE,     -- Actual business date (3 days ago)
    RECORD_SOURCE,
    AMOUNT
)
```

---

### 💡 Key Learning — Late Data Toolkit

| Solution                         | Layer           | When to Use                           |
| -------------------------------- | --------------- | ------------------------------------- |
| **Reprocessing window**          | Batch pipelines | Always process last N days            |
| **MERGE / Upsert**               | DWH layer       | Insert or update late records         |
| **Watermarking**                 | Streaming layer | Define tolerance for late events      |
| **Idempotent design**            | All layers      | Safe to rerun without duplicates      |
| **Dual timestamps in Raw Vault** | Data Vault      | Separate load time from business time |
