### Q2. Batch vs Streaming — Lambda vs Kappa Architecture

**Lambda Architecture:**
- Has **two paths**: Batch layer + Speed (stream) layer
- Batch layer: reprocesses all data for accuracy
- Speed layer: processes recent data for low latency
- Serving layer: merges both outputs

```
Raw Data → [Batch Layer]  → Batch Views  ↘
         → [Speed Layer] → Real-time Views → [Serving Layer] → Query
```

**Kappa Architecture:**
- Has **one path**: Stream only
- All data (real-time + historical) goes through the streaming pipeline
- Simpler to maintain — no dual codebase

```
Raw Data → [Stream Processing Layer] → [Serving Layer] → Query
```

**Which to choose?**

For this fraud detection system → **Kappa Architecture**

- Latency requirement is < 2 seconds → streaming is the primary need
- Reprocessing can be handled via Kafka retention + replay
- No need to maintain two separate pipelines
- Simpler ops, less code duplication

---

### Q3. How to support reprocessing of historical data?

**Design approach:**

1. **Kafka Retention** — Keep raw events in Kafka for 7–30 days
   - If pipeline has a bug, replay from Kafka offset
2. **Data Lake as source of truth** — Write all raw events to S3/GCS immediately
   - Reprocess from the lake using Spark batch jobs
3. **Idempotent pipelines** — Use unique transaction IDs as keys
   - Re-running the pipeline with same data produces same output (no duplicates)

```
Kafka (30-day retention)
    ↓
S3 Raw Zone (permanent storage)
    ↓
Spark Batch Reprocessing (triggered on demand)
    ↓
Overwrite results in Warehouse / Delta table
```

**Key principle:** `event_id` as deduplication key ensures idempotency.

## ⚡ PART 2 — Spark Deep Dive

### Q1. Spark Architecture

#### Components

| Component | Role |
|---|---|
| **Driver** | Main process — hosts SparkSession, builds DAG, coordinates execution |
| **Executor** | Worker processes — runs tasks, caches data, reports back to driver |
| **Cluster Manager** | Allocates CPU + memory (YARN / Kubernetes / Standalone) |
| **DAG Scheduler** | Splits job into **stages** at shuffle boundaries |
| **Task Scheduler** | Splits stages into **tasks**, assigns to executors |

#### Job Execution Flow

```
User Code (PySpark)
    ↓
Driver creates Logical Plan
    ↓
Converted to Physical Plan
    ↓
DAG Scheduler → splits into Stages (boundary = shuffle)
    ↓
Task Scheduler → splits Stage into Tasks (1 task per partition)
    ↓
Tasks sent to Executors
    ↓
Executors process partitions
    ↓
Results returned to Driver
```

#### Example: `spark.read.parquet().groupBy().count()`

```
Stage 1: read parquet → (narrow) filter/map
Stage 2: groupBy → shuffle → count   ← stage boundary here
```

### Q2. Narrow vs Wide Transformations

#### Narrow Transformation
- **1 input partition → 1 output partition**
- No shuffle needed
- Examples: `select()`, `filter()`, `map()`, `withColumn()`

```
Partition 1 → Partition 1
Partition 2 → Partition 2
Partition 3 → Partition 3
```

#### Wide Transformation
- **Multiple input partitions → multiple output partitions**
- Requires **shuffle** (data redistribution across network)
- Examples: `groupBy()`, `join()`, `reduceByKey()`, `distinct()`

```
Partition 1 ↘
Partition 2 → [Shuffle] → New Partitions
Partition 3 ↗
```

#### Why is Shuffle Expensive?
Shuffle involves all of:
- 💾 **Disk I/O** — write shuffle data to disk
- 🌐 **Network transfer** — move data across nodes
- 🔄 **Serialization/Deserialization**
- 🔢 **Sorting**

- ✅ Correct definition and examples
- ✅ Correct reasoning on why shuffle is expensive

### Q3. Partitioning Strategy

#### Rule of Thumb
```
Ideal partition size = 128MB
```

#### Calculation for 1TB dataset
```
1 TB = 1024 GB = 1,048,576 MB
Partitions = 1,048,576 / 128 ≈ 8,192 partitions
```

#### Too Small Partitions (e.g., 10MB)
- Too many tasks created
- **Scheduler overhead** dominates
- Spark wastes time scheduling instead of computing

#### Too Large Partitions (e.g., 2GB)
- **Executor memory pressure** → OOM errors
- Long-running tasks (no parallelism)
- Risk of **data skew**

### Q4. Broadcast Join vs Shuffle Join

#### Shuffle Join (Default)
- Both tables are **shuffled** across the network
- Matching keys land on same executor → joined
- **Expensive** for large tables

```
Table A (1B rows) → Shuffle
Table B (100M rows) → Shuffle
                        ↓
                   Join on same partition
```

**Problem:** Heavy network I/O, disk writes, slow.

#### Broadcast Join
- Small table is **copied (broadcasted) to every executor**
- Large table stays in place
- Join happens **locally** — no shuffle!

```
Table A (1B rows) → stays on executors
Table B (200 rows) → broadcast to ALL executors
                        ↓
                   Local join on each executor
```

#### When does Spark auto-broadcast?
```python
# Default threshold
spark.sql.autoBroadcastJoinThreshold = 10MB  # (10 * 1024 * 1024 bytes)
```
If a table is < 10MB, Spark automatically uses broadcast join.

#### Manual Broadcast
```python
from pyspark.sql.functions import broadcast

df.join(broadcast(dim_table), "user_id")
```

#### When to use which?

| Scenario | Join Type |
|---|---|
| Both tables large (GB/TB) | Shuffle Join |
| One table small (< 10MB by default) | Broadcast Join |
| Dimension table join | Broadcast Join |
| Fact-to-fact join | Shuffle Join |

### Q5. Spark Optimization Techniques (8 techniques)

| # | Technique | How |
|---|---|---|
| 1 | **Partition Tuning** | Set `spark.sql.shuffle.partitions` appropriately; use `repartition()` |
| 2 | **Broadcast Join** | Broadcast small tables to avoid shuffle |
| 3 | **Caching** | `df.cache()` for datasets reused multiple times |
| 4 | **Predicate Pushdown** | Filter early; Spark pushes `WHERE` clause to data source (Parquet/Delta) |
| 5 | **Column Pruning** | Use `SELECT col1, col2` instead of `SELECT *` |
| 6 | **File Size Optimization** | Aim for 100MB–1GB files; avoid small files problem |
| 7 | **Data Partitioning** | Partition by common filter columns (e.g., `year/month/day`) |
| 8 | **Bucketing** | `bucketBy(100, "user_id")` — reduces shuffle during repeated joins |

#### Example — Predicate Pushdown
```python
# Spark pushes this filter down to Parquet reader
# Only reads relevant row groups — not entire file
df = spark.read.parquet("s3://bucket/transactions") \
    .filter("date = '2025-01-01'")
```

#### Example — Caching
```python
# Without cache: computed twice
df.count()
df.show()

# With cache: computed once, reused
df.cache()
df.count()
df.show()
```
