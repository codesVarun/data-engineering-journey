## 📝 Interview Q&A — My Answers (Refined)

**Q1: How do we calculate delivery time?**

> Delivery time = timestamp of `DELIVERED` event − timestamp of `ORDER_PLACED` event for the same `order_id`. We use `MIN(event_time)` for ORDER_PLACED and `MAX(event_time)` for DELIVERED to handle duplicate events.

**Q2: Why do we need event_id?**

> `event_id` ensures uniqueness of each event and enables deduplication in the Silver layer. Since Kafka is at-least-once delivery, the same message can arrive twice — `event_id` guarantees exactly-once processing.

**Q3: What happens if Spark crashes?**

> Spark recovers using **checkpointing**. Kafka retains data, and Spark resumes from the last committed offset stored in the checkpoint, ensuring no data loss and no duplicate reprocessing. This is the exactly-once semantics story: Kafka = durability, Spark checkpoint = recovery.

**Q4: Why not write directly to Snowflake without Kafka?**

> Kafka decouples producers from consumers. Without it: systems are tightly coupled, Snowflake can't handle high-frequency writes efficiently, there's no replay on failure, and multiple consumers can't independently consume the same stream. Kafka enables event-driven architecture that Snowflake alone cannot support.

**Q5: Why partition by order_id in Kafka?**

> Partitioning by `order_id` ensures all events for a single order go to the same partition, preserving sequential ordering (`ORDER_PLACED → ORDER_ACCEPTED → PICKED_UP → DELIVERED`). Kafka guarantees ordering **only within a partition** — this is critical for correct metric computation.

**Q6: What problem does watermarking solve?**

> Watermarking handles late-arriving events in streaming. It defines how long Spark waits for late data before finalizing aggregations — ensuring correctness without waiting forever and keeping state bounded. Example: if `DELIVERED` arrives 10 minutes late, watermark absorbs it; without it, metrics would be wrong.

**Q7: Why not go Bronze → Gold directly (skipping Silver)?**

> Bronze contains raw, unvalidated data — not safe for business use. Silver enforces data quality: deduplication, type casting, null handling, schema validation. Gold then applies business logic on top of a trusted, clean dataset. Skipping Silver means complex logic in one step and no reusable clean layer for other teams. **Medallion architecture enforces data quality contracts at each stage.**
