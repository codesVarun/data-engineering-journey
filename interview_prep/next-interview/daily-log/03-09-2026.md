### Question 1 — Latest Event Per Policy

**Table: `policy_events`**

| policy_id | event_time | status |
|---|---|---|
| P1 | 2024-01-01 | Active |
| P1 | 2024-02-01 | Active |
| P1 | 2024-03-01 | Cancelled |
| P2 | 2024-01-10 | Active |
| P2 | 2024-02-20 | Expired |

**Task:** Return the latest event per policy.

#### ✅ Correct Solution

```sql
WITH ranked_events AS (
    SELECT
        policy_id,
        event_time,
        status,
        ROW_NUMBER() OVER (
            PARTITION BY policy_id
            ORDER BY event_time DESC
        ) AS rn
    FROM policy_events
)
SELECT policy_id, event_time, status
FROM ranked_events
WHERE rn = 1;
```

#### ❌ My Attempt

```sql
Select policy_id, event_time OVER (PARTITION BY policy_id ORBER BY event_time DESC), status
from policy_events;
```

#### Problems Identified

| # | Problem |
|---|---|
| 1 | `OVER()` used without a window function |
| 2 | `ORDER` misspelled as `ORBER` |
| 3 | Missing `ROW_NUMBER()` |
| 4 | No filtering for latest record (`WHERE rn = 1`) |

#### 💡 Key Learning

To select the latest record per group:
- Use a window function
- `PARTITION BY` the grouping column
- `ORDER BY` timestamp `DESC`
- Filter `WHERE ROW_NUMBER() = 1`

> This is a very common **data engineering deduplication pattern** — memorize it.

### Question 2 — Running Total Per Policy

**Table: `transactions`**

| policy_id | transaction_date | amount |
|---|---|---|
| P1 | 2024-01-01 | 100 |
| P1 | 2024-02-01 | 200 |
| P1 | 2024-03-01 | 150 |
| P2 | 2024-01-05 | 300 |

**Task:** Calculate running total per policy ordered by transaction date.

#### ✅ Correct Solution

```sql
SELECT
    policy_id,
    transaction_date,
    amount,
    SUM(amount) OVER (
        PARTITION BY policy_id
        ORDER BY transaction_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total
FROM transactions;
```

#### ❌ My Attempt

```sql
SELECT policy_id, transaction_date, amount,
    SUM(amount) OVER (PARTITION BY policy_id ORDER BY transaction_date DES) AS running_total
FROM transactions;
```

#### Problems Identified

| # | Problem |
|---|---|
| 1 | `DESC` misspelled as `DES` |
| 2 | Running totals should use ascending order (oldest → newest) |
| 3 | Window frame not explicitly defined (`ROWS BETWEEN ...`) |

#### 💡 Key Learning

Running totals require:

```sql
SUM(column) OVER (
    PARTITION BY group_column
    ORDER BY time_column
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
)
```

Used frequently in: financial reports, transaction analysis, cumulative metrics.

### Question 3 — Top 2 Claims Per Policy (Including Ties)

**Table: `claims`**

| policy_id | claim_id | claim_amount |
|---|---|---|
| P1 | C1 | 1000 |
| P1 | C2 | 900 |
| P1 | C3 | 900 |
| P1 | C4 | 700 |
| P2 | C5 | 1200 |
| P2 | C6 | 1000 |

**Task:** Return top 2 claim amounts per policy including ties.

#### ✅ Correct Solution

```sql
WITH ranked_claims AS (
    SELECT
        policy_id,
        claim_id,
        claim_amount,
        DENSE_RANK() OVER (
            PARTITION BY policy_id
            ORDER BY claim_amount DESC
        ) AS rn
    FROM claims
)
SELECT policy_id, claim_id, claim_amount
FROM ranked_claims
WHERE rn <= 2;
```

#### ❌ My Attempt

```sql
WITH ranked_claims (SELECT policy_id, claim_id, claim_amount,
    DENSE_RANK OVER (PARTITION BY policy_id ORDER BY claim_amount DES) AS RN
    FROM claims)
SELECT * FROM ranked_claims WHERE RN >= 2;
```

#### Problems Identified

| # | Problem |
|---|---|
| 1 | Missing `AS` keyword in CTE definition |
| 2 | Missing parentheses on `DENSE_RANK()` |
| 3 | `DESC` misspelled as `DES` |
| 4 | Logic wrong — `RN >= 2` returns rank 2 and above, should be `RN <= 2` |

#### 💡 Key Learning

Use `DENSE_RANK()` when **ties must be included** in top-N queries:

```sql
DENSE_RANK() OVER (PARTITION BY column ORDER BY value DESC)
```

| Function | Behavior on Ties |
|---|---|
| `ROW_NUMBER()` | Assigns unique rank — ties get different numbers |
| `RANK()` | Ties get same rank, next rank is skipped |
| `DENSE_RANK()` | Ties get same rank, next rank is NOT skipped ✅ |

## 🏗️ Case Study 1 — Insurance Data Pipeline Design

**Requirements:**
- 200M+ records daily
- Preserve full history
- Analytics and reporting
- Real-time + batch ingestion

### ✅ Correct Architecture

```
Source Systems
      ↓
Streaming / Batch Ingestion (Kafka / Snowpipe)
      ↓
AWS S3 — Data Lake (Raw Landing)
      ↓
Spark Processing (Transformation & Cleansing)
      ↓
Snowflake Data Warehouse
      ↓
Data Vault 2.0 Model (Raw Vault → Business Vault → Data Mart)
      ↓
Business Intelligence / Reporting
```

### Role of Each Component

| Component | Role |
|---|---|
| **AWS S3** | Raw storage for all incoming data — cheap, durable, scalable |
| **Apache Spark** | Distributed processing, transformation, and cleansing |
| **Snowflake** | Cloud data warehouse optimized for analytical queries |
| **Data Vault 2.0** | Enterprise data modeling — full history, auditability, flexibility |

### Data Vault Structure

| Component | Stores | Example |
|---|---|---|
| **HUB** | Unique business keys only | `policy_id`, `customer_id` |
| **LINK** | Relationships between hubs | `policy ↔ customer` |
| **SATELLITE** | Descriptive attributes + history | `premium`, `status`, `coverage details` |

### ❌ Mistakes I Made

| Mistake | Correction |
|---|---|
| Said HUB contains business attributes | ❌ Wrong — HUB contains **business keys only**. Attributes go in Satellites. |
| Suggested separate PROD_CLONE warehouse for ingestion | ❌ Not a realistic or standard architecture pattern |

### 💡 Key Learning

Data Vault loading order always flows:

```
Raw Vault → Business Vault → Data Mart
```

Never mix business rules into the Raw Vault layer.

## 🏗️ Case Study 2 — Query Performance Problem

**Query:**

```sql
SELECT policy_id, SUM(amount)
FROM transactions
WHERE transaction_date BETWEEN '2024-01-01' AND '2024-12-31'
GROUP BY policy_id;
```

**Table size: 3 billion rows**

### Why This Query Might Be Slow

- Full table scan on 3B rows — no pruning
- Heavy aggregation across large dataset
- Warehouse size too small for the workload
- No clustering on `transaction_date`
- Poor partition pruning

### ✅ Snowflake Optimization Techniques

| Technique | How It Helps |
|---|---|
| **Micro-partition pruning** | Skips partitions that don't match the date filter |
| **Clustering keys** | Organizes data physically to improve pruning |
| **Columnar storage** | Only reads `transaction_date`, `policy_id`, `amount` — not entire rows |
| **Result caching** | Re-running same query returns instantly from cache |
| **Warehouse scaling** | Scale up warehouse size for heavy aggregation jobs |

### Applying Clustering Keys

```sql
ALTER TABLE transactions
CLUSTER BY (transaction_date, policy_id);
```

This physically organizes data by date — so a filter on `transaction_date` prunes the vast majority of micro-partitions.

### ❌ Mistakes I Made

| Mistake | Correction |
|---|---|
| Assumed Snowflake uses traditional indexing | ❌ Wrong — Snowflake has **no traditional indexes** |

**Snowflake instead uses:**
- Micro-partitions with metadata (min/max values per column)
- Clustering keys for manual organization
- Automatic query optimizer
- Result and metadata caching

### 💡 Key Learning

The 4 core Snowflake performance tools to always mention:

1. **Micro-partitions** — automatic, stores metadata for pruning
2. **Clustering keys** — manual, organizes large tables for filter-heavy queries
3. **Warehouse scaling** — more compute for heavy aggregation
4. **Result caching** — free performance for repeated queries
