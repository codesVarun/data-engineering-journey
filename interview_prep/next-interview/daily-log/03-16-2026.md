## 🧠 SQL Question 1 — Median Salary

### ❌ My Attempt (Issues)
```sql
WITH AS ordered_salary AS (SELECT id, salary, ROW_NUMBER() OVER (ORDER BY salary) AS rn FROM salary), 
WITH salary_with_total_ct AS (SELECT count(*) OVER (rn) AS total_count FROM ordered_salary)
```
**Issues:**
- `WITH AS` syntax is wrong
- `COUNT(*) OVER(rn)` is incorrect
- Median selection logic missing

### ✅ Correct Query
```sql
WITH ordered_salary AS (
    SELECT
        salary,
        ROW_NUMBER() OVER (ORDER BY salary) AS rn,
        COUNT(*) OVER () AS total_count
    FROM salary
)
SELECT AVG(salary) AS median_salary
FROM ordered_salary
WHERE rn IN (
    (total_count + 1) / 2,
    (total_count + 2) / 2
);
```

### 💡 Key Logic
| Case | Formula | Example (n=5) | Example (n=4) |
|------|---------|--------------|--------------|
| Odd  | (n+1)/2 and (n+2)/2 both = same row | Row 3 | — |
| Even | (n+1)/2 and (n+2)/2 = two middle rows | — | Rows 2 & 3 |

> `AVG()` handles both cases — returns single value for odd, averages two for even.

## 🧠 SQL Question 2 — Overlapping Meetings

### ❌ My Attempt (Issues)
```sql
SELECT t1.meeting_id, t2.meeting 
FROM meetings t1 
SELF JOIN meetings t2 
ON t1.meeting = t2.meeting_id 
WHERE t1.start < t2.end AND t2.start < t1.end;
```
**Issues:**
- `SELF JOIN` is not valid SQL syntax — use `JOIN`
- Wrong join condition (`ON t1.meeting = t2.meeting_id`)
- Column `t2.meeting` doesn't exist
- Would produce duplicate pairs (1,2) and (2,1)

### ✅ Correct Query
```sql
SELECT 
    t1.meeting_id AS meeting1,
    t2.meeting_id AS meeting2
FROM meetings t1
JOIN meetings t2
    ON t1.meeting_id < t2.meeting_id
   AND t1.start_time < t2.end_time
   AND t2.start_time < t1.end_time;
```

### 💡 Key Concepts
- `t1.meeting_id < t2.meeting_id` — avoids duplicate pairs
- **Overlap condition:** `start1 < end2 AND start2 < end1`

## 🧠 SQL Question 3 — First Purchase After Signup

### ❌ My Attempt (Issues)
```sql
SELECT u.user_id, o.order_date AS first_order_date, 
ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY order_date DESC) AS rn 
FROM users u 
LEFT JOIN orders o ON u.user_id = o.user_id 
WHERE rn = 1;
```
**Issues:**
- Window function cannot be used directly in `WHERE` clause
- `ORDER BY order_date DESC` gives *latest* order, not first
- Missing filter: order date must be after signup date

### ✅ Correct Query
```sql
WITH ranked_orders AS (
    SELECT
        u.user_id,
        o.order_date,
        ROW_NUMBER() OVER (
            PARTITION BY u.user_id
            ORDER BY o.order_date
        ) AS rn
    FROM users u
    JOIN orders o
        ON u.user_id = o.user_id
    WHERE o.order_date >= u.signup_date
)
SELECT
    user_id,
    order_date AS first_order_date
FROM ranked_orders
WHERE rn = 1;
```

## ⚙️ Data Engineering — Data Skew

### 1️⃣ What is Data Skew?
Data skew occurs when data distribution across partitions is uneven — some tasks process far more data than others.

**Example:**
```
policy_id A  → 300M rows  ❌ skewed
policy_id B  → 10K rows
policy_id C  → 8K rows
```

### 2️⃣ How it Affects Spark
Spark processes data per partition. A skewed partition creates a **straggler task**:
```
Executor 1 → 5M rows  → finishes in 30 sec
Executor 2 → 4M rows  → finishes in 25 sec
Executor 3 → 300M rows → runs for 40 min ❌
```
The entire job waits for the slowest partition.

### 3️⃣ 5 Ways to Fix Data Skew

| # | Technique | How |
|---|-----------|-----|
| 1 | **Salting** | Append random number to skewed key to spread rows across partitions |
| 2 | **Broadcast Join** | If one table is small, use `broadcast(small_table)` — avoids shuffle |
| 3 | **Repartition** | `df.repartition(500)` — manually increase partition count |
| 4 | **Skew Join (Spark 3+)** | Enable `spark.sql.adaptive.skewJoin.enabled` — auto splits large partitions |
| 5 | **Pre-aggregation** | `GROUP BY policy_id` before the join to reduce data volume |

### 4️⃣ Detecting Skew in Spark UI
- **Stage → Tasks → Duration:** One task taking 2 min vs others taking 5 sec
- **Shuffle Read Size:** One partition with disproportionately large shuffle read
- **Uneven partition sizes** in the metrics tab

## ⚙️ Star Schema vs Snowflake Schema

### 1️⃣ Star Schema
One central **Fact table** connected directly to **denormalized Dimension tables**.
```
         DIM_DATE
             |
DIM_PRODUCT — FACT_SALES — DIM_CUSTOMER
             |
         DIM_REGION
```

### 2️⃣ Snowflake Schema
Dimension tables are **normalized** — they link to further sub-dimension tables.
```
FACT_SALES → DIM_PRODUCT → PRODUCT_CATEGORY → PRODUCT_BRAND
```

### 3️⃣ Advantages

| | Star Schema | Snowflake Schema |
|---|------------|-----------------|
| **Speed** | ✅ Faster queries (fewer joins) | Slower (more joins) |
| **Simplicity** | ✅ Easier for BI tools | More complex |
| **Storage** | Uses more (denormalized) | ✅ Less storage |
| **Data Integrity** | Lower | ✅ Higher (normalized) |
| **Scalability** | Moderate | ✅ Better for complex domains |

### 4️⃣ Preferred in Warehouses
**⭐ Star Schema** — analytics workloads prioritize query speed over storage efficiency.

### 5️⃣ Real Project Usage
- **Insurance:** Snowflake schema — separate star schemas per subject area (Policy, Claims, Transactions) linked together
- **E-commerce:** Star schema — `FACT_ORDERS` with `DIM_CUSTOMER`, `DIM_PRODUCT`, `DIM_DATE`, `DIM_REGION`

## ⚠️ Weak Areas to Revisit
- CTE syntax: `WITH cte_name AS (...)` — no `AS` before the name
- Window functions **cannot** be used in `WHERE` — always wrap in a CTE first
- Self-join: use `JOIN ... ON t1.id < t2.id` to avoid duplicate pairs
- `ORDER BY ASC` for first/earliest records, `DESC` for latest
