## 📚 What I Learned

- **Running balance** — `SUM(amount) OVER (PARTITION BY id ORDER BY date)` accumulates row by row in chronological order
- **Latest record without window functions** — use `MAX(event_time) GROUP BY policy_id` in a subquery, then JOIN back to get the full row
- **Gaps & Islands trick** — `date - ROW_NUMBER()` gives the same constant value for consecutive dates; rows with same result = same streak group
- **Exactly Once Processing** — means each event is processed exactly one time, no duplicates, no skips
- **Why Exactly Once is hard** — distributed systems can crash mid-processing; on restart the same event may be processed again
- **How Spark + Kafka achieve it** — Kafka offsets (resume from last position) + Spark checkpointing (save state) + Idempotent writes (MERGE instead of INSERT)
- **SCD Type 2** — creates a new row for every change instead of updating in place; preserves full history
- **SCD2 columns** — `effective_start_date`, `effective_end_date`, `is_current` / `current_row_indicator`

## 🔑 Key Patterns to Memorise

### Gaps & Islands
```sql
WITH numbered AS (
    SELECT policy_id, transaction_date,
           ROW_NUMBER() OVER (
               PARTITION BY policy_id
               ORDER BY transaction_date
           ) AS rn
    FROM transactions
),
grouped AS (
    SELECT policy_id, transaction_date,
           transaction_date - rn * INTERVAL '1 day' AS grp
    FROM numbered
)
SELECT policy_id,
       MIN(transaction_date) AS start_date,
       MAX(transaction_date) AS end_date
FROM grouped
GROUP BY policy_id, grp;
```

### Latest Record Without Window Functions
```sql
SELECT e.policy_id, e.event_time, e.status
FROM events e
JOIN (
    SELECT policy_id, MAX(event_time) AS latest_time
    FROM events
    GROUP BY policy_id
) m ON e.policy_id = m.policy_id
   AND e.event_time = m.latest_time;
```
